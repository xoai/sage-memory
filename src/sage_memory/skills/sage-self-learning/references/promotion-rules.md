# Promotion Rules

Learnings start at project scope. When they prove broadly applicable,
they escalate — first to global scope, then to shared team files. This
document defines when and how escalation happens.

## Promotion Ladder

```
Project scope (sage-memory project DB)
  ↓ Learning applies beyond this codebase
Global scope (sage-memory global DB)
  ↓ Learning would help other developers
Team files (committed to repo)
  ↓ Learning is critical enough for session-start context
CLAUDE.md / AGENTS.md (system prompt level)
```

Each step increases the audience and visibility. Each step also requires
higher confidence that the learning is correct and broadly applicable.

## Project → Global

### When to Promote

A project-scope learning should promote to global when ALL of these
are true:

1. **Context-independent.** The learning makes sense without knowing
   this specific project. "Stripe webhook requires raw body" is
   context-independent. "The billing service webhook handler is at
   /api/v2/webhooks" is not.

2. **Reusable.** You'd want to know this in a different project that
   uses the same library, tool, or pattern.

3. **Verified.** The learning has been confirmed — either by a
   successful fix, by user validation, or by observing the issue
   multiple times.

### How to Detect Promotion Candidates

**Recurrence signal:** When search-before-store finds 2+ similar
learnings across different projects, the pattern is recurring and
should be global.

**Type signal:** Some types are more likely to be global:
- `api-drift` → almost always global (library changes affect everyone)
- `error-fix` → often global (platform issues, common tool errors)
- `gotcha` → sometimes global (depends on whether it's library-specific
  or project-specific)
- `convention` → rarely global (conventions are inherently local)
- `correction` → rarely global (usually project-specific assumptions)

**User signal:** The user says "that's a common issue" or "everyone hits
this" or "good to know for future projects."

### How to Promote

1. **Rephrase for context-independence.** Remove project-specific
   details. Generalize file paths to patterns. Keep the fix concrete.

   Before (project scope):
   ```
   Title: [LRN:gotcha] Express body parser breaks Stripe webhook at /api/webhooks/stripe
   Content: ...our webhook handler at src/api/webhooks/stripe.ts...
   ```

   After (global scope):
   ```
   Title: [LRN:gotcha] Express body parser must be skipped for Stripe webhook verification
   Content: ...any Express route handling Stripe webhooks...
   ```

2. **Store at global scope:**
   ```
   sage_memory_store:
     title: "[LRN:gotcha] Express body parser must be skipped for Stripe webhook verification"
     content: "<rephrased, context-independent version>"
     tags: ["self-learning", "gotcha", "stripe", "webhooks", "express"]
     scope: "global"
   ```

3. **Keep the project-scope version.** Don't delete it — it has
   project-specific details (file paths, route names) that are useful
   in this project. The global version is a generalized sibling, not
   a replacement.

## Global → Team Files

### When to Export

Export validated learnings to shared files when ANY of these are true:

1. **The learning prevents a recurring team mistake.** Multiple
   developers (or sessions) have hit the same issue.

2. **The learning documents a team convention** that isn't written
   down anywhere else.

3. **The learning is critical for onboarding.** A new developer (human
   or agent) working on this codebase would need to know this.

4. **The user asks.** "Add this to our team docs" or "make sure
   everyone knows this."

### Where to Export

| Content type | Export target | When |
|---|---|---|
| Project conventions | `CLAUDE.md` or `AGENTS.md` | Convention learnings that any agent should know |
| Build/setup gotchas | `CLAUDE.md` § Build & Dependencies | Setup issues that block new contributors |
| API-specific gotchas | `.sage/team-learnings.md` | Collection of learned gotchas for this codebase |
| Workflow patterns | `AGENTS.md` | Multi-step processes the agent should follow |

### Export Format

Learnings exported to team files should be **distilled into concise
rules**, not verbose incident reports.

**In `.sage/team-learnings.md`:**
```markdown
# Team Learnings

Gotchas, conventions, and patterns discovered through development.
Auto-generated by self-learning — edit freely.

## Build & Setup

- **pnpm only.** This project uses pnpm workspaces. Do not run
  `npm install`. Check for `pnpm-lock.yaml`.

## API Gotchas

- **Stripe webhooks: skip body parser.** Signature verification
  requires the raw body. Use `express.raw()` for webhook routes.

## Conventions

- **API response envelope.** All handlers return `{data, error, meta}`.
  Exception: webhook handlers return raw status codes.
```

**In `CLAUDE.md`** (one-liners, system-prompt level):
```markdown
## Build & Dependencies
- Package manager: pnpm (not npm) — use `pnpm install`
- Stripe webhooks require raw body — skip Express body parser
```

### Export Process

1. Select learnings that meet the export criteria.
2. Distill each into a concise rule (1-3 lines, not the full learning).
3. Group by category (build, API, conventions, etc.).
4. Write to the appropriate file. Create the file if it doesn't exist.
5. Inform the user: "I've exported 3 learnings to .sage/team-learnings.md
   for the team. Would you like to review before committing?"

## Team Files → CLAUDE.md

The highest promotion tier. CLAUDE.md is read at session start before
any MCP tools are available. Only the most critical learnings belong
here — things that affect every task, not just specific scenarios.

### Criteria for CLAUDE.md Promotion

ALL must be true:

1. **Affects every session.** Not "when working with Stripe" but "always
   use pnpm, not npm."
2. **Prevents immediate failure.** Getting this wrong would block
   progress or cause errors in the first few minutes.
3. **Can be expressed in one line.** If it takes a paragraph to explain,
   it belongs in sage-memory or team-learnings.md, not CLAUDE.md.

### When NOT to Promote to CLAUDE.md

- Library-specific gotchas (belongs in sage-memory, surfaced by recall)
- Detailed debugging insights (too long for system prompt)
- Learnings that only apply to one module or feature
- Anything already covered by existing CLAUDE.md content
