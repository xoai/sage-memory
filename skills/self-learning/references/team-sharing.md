# Team Sharing

How to share learnings beyond a single developer. Three levels of
sharing, from lightweight to structured.

## Level 1: Shared Files in Repo (v0.1)

The simplest sharing mechanism. Export learnings to human-readable
markdown files that get committed alongside code.

### File: `.sage/team-learnings.md`

The primary export target. A living document of team-accumulated
gotchas, conventions, and patterns.

```markdown
# Team Learnings

Gotchas, conventions, and patterns discovered through development.
Generated by self-learning. Edit freely — this is a team resource.

Last updated: 2026-03-17

## Build & Setup

- **Package manager: pnpm.** Do not use npm. Lock file is
  `pnpm-lock.yaml`. Run `pnpm install` for dependencies.

- **Node version: 20+.** The project uses Node 20 features.
  `.nvmrc` is present — run `nvm use` before starting.

## API Gotchas

- **Stripe webhooks: raw body required.** Signature verification
  needs the raw request body. Skip Express body parser for webhook
  routes using `express.raw({type: 'application/json'})`.

- **Auth tokens: refresh silently.** The auth service returns 401
  with a `X-Token-Expired: true` header when tokens expire. Retry
  once with a refreshed token before failing.

## Conventions

- **API response envelope.** All handlers in `src/api/` return
  `{data, error, meta}`. Validation errors use 422 with error field.
  Exception: webhook handlers return raw status codes.

- **Test fixtures: module-scoped.** Database fixtures in `conftest.py`
  use module scope for performance. Don't create function-scoped
  fixtures for DB setup.

## Debugging Tips

- **504 on /api/reports: check N+1 queries.** The reports endpoint
  is sensitive to N+1 issues. Use `prefetch_related()` for any new
  querysets in ReportBuilder.
```

### How Another Agent Consumes This

Other developers' agents can consume `.sage/team-learnings.md` through
two channels:

1. **Direct file reading.** At session start, read
   `.sage/team-learnings.md` if it exists. This happens naturally as
   part of the memory skill's "Read the Room" phase, or the agent can
   check for it during self-learning recall.

2. **sage-memory import (future).** When the agent has sage-memory
   available, it could parse team-learnings.md and import entries into
   its project-scope memory. This enables search-based recall instead
   of reading the full file. Deferred to v1.1.

### Keeping the File Current

The export file can grow stale as code changes. Practices to keep it
fresh:

- **Export appends, doesn't replace.** New learnings are added to the
  relevant section. Existing entries stay unless explicitly removed.

- **Review on major changes.** When a module is refactored or a
  dependency is updated, review the related learnings. Remove entries
  that no longer apply.

- **User controls export.** The agent suggests exports but doesn't
  write to team files without confirmation. "I have 2 learnings that
  might help the team. Want me to add them to .sage/team-learnings.md?"

## Level 2: Git-Tracked Learnings (v1.1 — Future)

A more structured format for teams that want machine-readable learnings
alongside the human-readable markdown.

### File: `.sage/team-learnings.json`

```json
{
  "version": "1.0",
  "updated": "2026-03-17",
  "learnings": [
    {
      "title": "[LRN:gotcha] Stripe webhook requires raw body",
      "content": "Signature verification needs raw body...",
      "type": "gotcha",
      "tags": ["stripe", "webhooks", "express"],
      "scope": "project",
      "added": "2026-03-15",
      "added_by": "agent"
    }
  ]
}
```

**Import workflow:** At session start, the agent reads
`.sage/team-learnings.json` and imports entries into sage-memory's
project DB. sage-memory's content-hash dedup prevents duplicates.
New entries (not yet in memory) get imported automatically.

**Export workflow:** When the agent promotes a learning to team scope,
it adds an entry to the JSON file. The file gets committed with the
next PR.

This gives teams: version-controlled learnings, machine-readable format,
automatic import into each developer's sage-memory, and human-reviewable
diffs in PRs.

## Level 3: Shared sage-memory Database (v2 — Future)

For organizations that want a centralized learning database.

Possible approaches:
- A shared sage-memory instance on a network path
- A sage-memory "sync" mechanism that merges entries across instances
- Integration with Context Hub for publishing validated learnings

This level requires sage-memory changes and is out of scope for v0.1.
Documented here to show the design direction.

## Sharing Across Scopes: Summary

| Scope | Who benefits | Mechanism | Version |
|-------|-------------|-----------|---------|
| Project | You, this project | sage-memory project DB | v0.1 |
| Global | You, all projects | sage-memory global DB | v0.1 |
| Team (file) | Team, this project | `.sage/team-learnings.md` in repo | v0.1 |
| Team (structured) | Team, this project | `.sage/team-learnings.json` + import | v1.1 |
| Organization | Company-wide | Shared DB or Context Hub | v2 |

## Privacy Considerations

When exporting learnings to shared files:

- **Strip sensitive data.** Error messages may contain API keys, tokens,
  internal URLs, or PII. Export the *insight*, not the raw error output.

- **Generalize file paths.** Use patterns (`src/api/webhooks/`) rather
  than developer-specific absolute paths.

- **Respect .gitignore.** If `.sage/` is in .gitignore, team sharing via
  repo files won't work. The agent should check and inform the user.

- **Ask before sharing.** The agent suggests exports but always asks for
  confirmation before writing to shared files.
